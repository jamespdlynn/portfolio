Atypical of the industry, I design my games to be generally server authoritative. This means the server does not simply act as a data relay, but is also responsible for executing a large portion of each game's business logic. This has a number of advantages over a codebase that exists nearly entirely on the client:

  • Lower likelihood of platform specific bugs (as all server code runs on the same Amazon Linux operating systems).
  • More stable integration tests with wider coverage.
  • Having validation reside on the server makes it much more difficult for players to cheat or fake purchases.
  • Server code changes can be deployed to production almost instantaneously, while it can take over a week to fully roll out a new version of a client to each app store.

I desire my server-side infrastructure to have a focus on low-latency and high-throughput request handling, while keeping overall costs to a minimum. In order to achieve this, I manage the entirety of my own custom stack within an AWS Virtual Private Cloud.

All server code is written in NodeJS and deployed using Elastic Beanstalk. Client API requests are load balanced between 2-6 EC2 service instances (number of instances are auto scaled based on current average latency and CPU utilisation). API Requests, along with being SSL encrypted, require expiratory custom authentication header tokens to ensure the security of each user's account and data.

The primary database is a MongoDB NoSQL cluster managed by MongoDB Atlas (though still hosted within the same AWS VPC as the rest of the stack). Two Elasticache Redis replica instances help reduce primary database load, by functioning as a temporary shared in-memory cache.

A private S3 bucket is used to store a game's remote Unity Asset Bundles (compressed batches of binary assets like textures and audio files). These are automatically propagated to CloudFront's global distribution network nodes, to be downloaded by their associated client devices.

Every client request, whether handled by CloudFront or the EC2 instances, has its pertinent information logged to CloudWatch for analytics and debugging. Logs are retained for up to 14 days.

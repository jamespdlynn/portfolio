Atypical of the industry, I design my games to be relatively server authoritative, meaning the server does not simply act as a data relay, but is also responsible for executing a large portion of each game's business logic. To do this effeciently, I manage the entirety of own custom server stack within an AWS Virtual Private Cloud.

Server authoritative applications have a number of advantages over ones where nearly all the code exists on the client:

  • As all the server code runs on the same Linux operating systems, there's less likelihood of encountering platform specific bugs.
  • Having validation reside on the server makes it much more difficult for players to cheat or fake purchases.
  • Server code is (generally) easier to write integration tests for.
  • Server code changes can be deployed to production almost instantaneously, while it can take over a week to roll out a new version of a client to all the app stores.

I designed my AWS stack with a focus on low-latency and high-throughput, while keeping overall costs to a minimum.

The server code is written in NodeJS and deployed using Elastic Beanstalk. All incoming client API requests are load balanced between 2-8 EC2 service instances (instances are auto scaled based on current average latency and CPU utilisation). API Requests, along with being SSL encrypted, require multiple types of custom authentication header token to ensure the security of each user's account and data.

The primary database is a MongoDB NoSQL cluster managed by MongoDB Atlas (though still hosted within the same AWS VPC as the rest of the stack). Two Elastic Cache Redis replica instances help reduce primary database load, by functioning as a temporary shared in-memory cache.

A private S3 bucket is used to store a game's remote Unity Asset Bundles (compressed batches of binary assets like textures and audio files). These are automatically propagated to CloudFront's global distribution network nodes, to be downloaded by their associated client devices.

Every client request, whether handled by Cloudfront or the EC2 instances, has its pertinent information logged to CloudWatch for analytics and debugging. Logs are retained for up to 14 days.
